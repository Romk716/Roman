# Пользовательские сценарии модуля «Сервис задач»

## Роли
- **Пользователь** — человек, который работает с модулем через какой-либо интерфейс (консольный, графический или API).
- **Система** — сам модуль, который выполняет операции.

## Список сценариев

### UC-1: Создание новой задачи
- **Цель:** Пользователь хочет добавить новую задачу.
- **Условия:** Модуль запущен, файл данных доступен.
- **Шаги:**
  1. Пользователь вводит название задачи (например, «Купить молоко») и описание (например, «1 литр»).
  2. Пользователь подтверждает создание.
- **Результат:** Создана задача с уникальным ID, статус «новая», задача отображается в общем списке и сохраняется в JSON-файл.
- **Приоритет:** критический

### UC-2: Просмотр списка всех задач
- **Цель:** Пользователь хочет увидеть все свои задачи.
- **Условия:** В модуле есть несколько задач.
- **Шаги:**
  1. Пользователь запрашивает список всех задач.
- **Результат:** Модуль возвращает массив всех задач с их полями.
- **Приоритет:** критический

### UC-3: Изменение статуса задачи
- **Цель:** Пользователь отмечает задачу как выполненную.
- **Условия:** Существует задача с ID=1.
- **Шаги:**
  1. Пользователь указывает ID задачи (1) и новый статус «завершена».
  2. Модуль применяет изменение.
- **Результат:** Статус задачи изменён на «завершена», данные сохранены в JSON.
- **Приоритет:** критический

### UC-4: Поиск задач по ключевому слову
- **Цель:** Пользователь ищет задачи, содержащие слово «отчёт».
- **Условия:** Есть задачи со словом «отчёт» в названии или описании.
- **Шаги:**
  1. Пользователь вводит ключевое слово «отчёт».
- **Результат:** Модуль возвращает список задач, где в title или description встречается «отчёт» (регистронезависимо).
- **Приоритет:** важный

### UC-5: Загрузка задач из файла при старте
- **Цель:** При запуске модуль восстанавливает ранее сохранённые задачи.
- **Условия:** Файл `tasks.json` существует и содержит корректные данные.
- **Шаги:**
  1. Пользователь запускает модуль.
- **Результат:** Модуль загружает задачи из файла, они доступны для работы.
- **Приоритет:** критический

### UC-6: Отсутствие файла при запуске
- **Цель:** Модуль корректно обрабатывает ситуацию, когда файл ещё не создан.
- **Условия:** Файл `tasks.json` отсутствует.
- **Шаги:**
  1. Пользователь запускает модуль.
- **Результат:** Модуль работает с пустым списком задач, не выдаёт ошибок. При создании первой задачи файл создаётся.
- **Приоритет:** важный

### UC-7: Повреждённый файл при запуске
- **Цель:** Модуль не падает, если файл повреждён.
- **Условия:** Файл `tasks.json` существует, но содержит некорректный JSON (например, синтаксическую ошибку).
- **Шаги:**
  1. Пользователь запускает модуль.
- **Результат:** Модуль определяет повреждение, начинает работу с пустым списком. При сохранении (например, создании задачи) файл перезаписывается корректными данными.
- **Приоритет:** важный

### UC-8: Попытка создать задачу без названия
- **Цель:** Модуль отклоняет создание задачи с пустым названием.
- **Условия:** Модуль запущен.
- **Шаги:**
  1. Пользователь пытается создать задачу, оставив название пустым.
- **Результат:** Модуль возвращает сообщение об ошибке, задача не создаётся.
- **Приоритет:** важный

### UC-9: Попытка установить недопустимый статус
- **Цель:** Модуль отклоняет изменение статуса на неразрешённое значение.
- **Условия:** Существует задача с ID=2, её текущий статус «новая».
- **Шаги:**
  1. Пользователь пытается установить статус «отложено» (не в списке допустимых).
- **Результат:** Модуль возвращает ошибку, статус остаётся прежним («новая»).
- **Приоритет:** дополнительный

### UC-10: Фильтрация задач по статусу
- **Цель:** Пользователь хочет увидеть только задачи в статусе «в работе».
- **Условия:** Есть задачи с разными статусами.
- **Шаги:**
  1. Пользователь запрашивает список задач со статусом «в работе».
- **Результат:** Модуль возвращает только те задачи, у которых статус «в работе».
- **Приоритет:** важный
